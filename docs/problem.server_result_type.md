## ServerをUIで使う際の責務境界について

### UIの責務は妥当か？
このプロダクトでは、最初の実装速度を極端に早めた。これは、ProviderやBoundaryなどの設計を待たずにデータ取得を実装することを意味する。
すると、高確率でServiceのような層からデータを拝借する構造が出来上がる。

**Service～UIの空白を埋めるのはServiceかUIのどちらか？**
このまま設計を押し進めるとすれば、必ずこの考えに至る。
Serviceにどこまで例外処理を任せるか？
UIはどこまでの情報を扱うのか？

現時点では、どちらにどれだけ寄せても何とかなってしまう。
扱うデータはProductsだけだから大丈夫と思ってしまう。

しかし、このプロダクトでは「BEが嘘を吐く」
今のまま責務境界を曖昧にしていれば必ず何処かで例外処理が分からなくなる。

### Serviceの返り値に滲む責務

ServiceはAPI通信を行う層として作られたが、UI都合や極端な設計保留によって責務過多になっている。
fetchデータを例外処理し、致命的エラーをthrowている。しかし、ここにProductsデータ整合検査まで付け加えられようとしている。

{ ok: boolean, error: Error, value: Product[] } にまとめるには、見かけ以上にやることが多い。
fetch結果の読み取り。エラーメッセージの存在確認と解釈。返り値に含める値の判断。Product[]型であることの確認。
これらは全てUIまでの道のりをここだけで埋めようとしている結果である。

返り値がこのまま固定されてしまえば拡張性は失われる。ここが最初の改修ポイントである。
